Toronto 311 Service Request Prioritization

Problem Statement

The City of Toronto receives approximately 1 million 311 service requests annually - from pothole repairs to wildlife emergencies. Traditional rule-based prioritization systems fail to adapt to seasonal surges, geographic demand shifts, and real-time backlog pressure, leading to delayed response times for critical incidents.

Reactive, Not Predictive Static rule-based systems cannot adapt to fluctuating demand, seasonal surges, or backlog pressure.
Seasonal Surge Blindness Volume increases ~40% between Q1 and Q3. A Feb 2025 snowstorm caused a 225% surge vs the prior year with no early warning.
Class Imbalance Problem Only 3.6% of requests are Open/In Progress (high priority). Accuracy alone is misleading - F1, Recall, and ROC-AUC are the critical metrics.
Uneven Geographic Demand Top 4 wards account for a disproportionate share of all requests. A one-size-fits-all strategy wastes limited city resources.

Business Questions driving this project:

  • Can historical 311 data predict request urgency or expected resolution delay?
  • Which factors most influence service prioritization - type, time, location, or backlog?
  • Does incorporating system-level context (backlog) improve prioritization accuracy?

Methodology - Four-Stage Pipeline

A structured pipeline was designed from raw data to dynamic priority scoring.

1
Data Cleaning & EDA
Remove duplicates, parse dates, fill missing values, explore temporal, geographic, and type-level demand patterns
2
Feature Engineering
Cyclical time encoding (sin/cos), frequency encoding for ward and service type, 3-day and 7-day rolling backlog windows
3
PCA & Modeling
Reduce 8 correlated features into 3 PCA components (63.5% variance). Train Logistic Regression, Random Forest, XGBoost
4
Priority Scoring
Hybrid score: 70% ML probability + 30% 7-day backlog. Rank all requests dynamically and compare vs rule-based approach

Data Overview & Preprocessing

Source: City of Toronto Open Data Portal - 311 Service Requests 2024-2025 (~1 million records, 44+ wards, 3.6% high priority).

  • Dropped duplicates and standardized all column names to snake_case
  • Dropped rows missing creation_date or service_request_type
  • Filled missing ward values with 'Unknown'; missing status with 'Open'
  • Parsed dates and extracted hour, day of week, month, and year
  • Grouped rare service types into 'other' - kept top 15 categories
  • Created is_peak_hour flag for 7-10 AM and 4-7 PM windows
  • Built daily_backlog as an intra-day cumulative request count
# Standardize column names df.columns = df.columns.str.lower().str.strip().str.replace(" ", "_") # Drop rows with critical missing values df = df.dropna(subset=["creation_date", "service_request_type"]) # Fill non-critical missing values df["ward"] = df["ward"].fillna("Unknown") df["status"] = df["status"].fillna("Open") # Group rare service types (keep top 15) top_services = df["service_request_type"].value_counts().nlargest(15).index df["service_request_type"] = df["service_request_type"].where( df["service_request_type"].isin(top_services), "other" ) # Build intra-day backlog counter df = df.sort_values("creation_date") df["daily_backlog"] = df.groupby(df["creation_date"].dt.date).cumcount()

Feature Engineering

Cyclical Time Encoding - Time is cyclical, not linear. Hour 23 and hour 0 are only 1 hour apart in reality but 23 units apart on a linear scale. Sin/cos encoding projects time onto a unit circle, preserving adjacency relationships and improving model performance on overnight patterns.

# Cyclical encoding - preserves circular nature of time df['hour_sin'] = np.sin(2 * np.pi * df['creation_hour'] / 24) df['hour_cos'] = np.cos(2 * np.pi * df['creation_hour'] / 24) df['month_sin'] = np.sin(2 * np.pi * df['creation_month'] / 12) df['month_cos'] = np.cos(2 * np.pi * df['creation_month'] / 12)

Rolling Backlog Features - Service requests are temporally dependent. A surge today increases the likelihood of backlog tomorrow. We constructed 3-day and 7-day rolling demand windows per service type.

df = df.sort_values(['service_request_type', 'creation_date']) def compute_rolling(group): group = group.copy().set_index('creation_date') group['rolling_3d_volume'] = group['status'].rolling('3D').count() group['rolling_7d_volume'] = group['status'].rolling('7D').count() return group.reset_index() df = df.groupby('service_request_type', group_keys=False).apply(compute_rolling)

PCA Dimensionality Reduction - Eight correlated engineered features were reduced to 3 orthogonal principal components, capturing 63.5% of total variance and eliminating multicollinearity before modeling.

from sklearn.preprocessing import StandardScaler from sklearn.decomposition import PCA pca_features = [ 'hour_sin', 'hour_cos', 'month_sin', 'month_cos', 'ward_freq', 'service_request_type_freq', 'rolling_3d_volume', 'rolling_7d_volume' ] X_scaled = StandardScaler().fit_transform(df[pca_features].fillna(0)) pca = PCA(n_components=3) df[['PC1', 'PC2', 'PC3']] = pca.fit_transform(X_scaled) # PC1: Demand & Backlog (34.3%) | PC2: Time-of-Day | PC3: Seasonal Pattern

Hybrid Priority Scoring

Rather than relying solely on ML probability, the model output was combined with real-time backlog context to produce a more operationally meaningful priority score.

# Hybrid score: 70% ML probability + 30% normalized 7-day backlog df['ml_priority_score'] = xgb.predict_proba(X)[:, 1] df['final_priority_score'] = ( 0.70 * df['ml_priority_score'] + 0.30 * df['rolling_7d_volume'] ) df['priority_rank'] = df['final_priority_score'].rank(ascending=False) # Comparison vs rule-based approach (top 1000 requests) ml_top = df.sort_values("final_priority_score", ascending=False).head(1000) rule_top = df.sort_values("service_request_type_freq", ascending=False).head(1000) print("Median Backlog - ML hybrid:", ml_top['rolling_7d_volume'].median()) print("Median Backlog - Rule-based:", rule_top['rolling_7d_volume'].median())
Why 70/30? The 70% ML component captures complex nonlinear patterns across service type, geography, and time. The 30% backlog component grounds the score in real operational pressure - ensuring requests submitted during surge periods are ranked higher, even when the ML score alone would not flag them.

Key Findings & Recommendations

  • XGBoost is the best model - F1 of 0.251, ROC-AUC of 0.930, PR-AUC of 0.428, outperforming both baseline models
  • Feature engineering doubled Recall and F1 - Rolling backlog features were the single most important predictor of urgency
  • Accuracy is misleading - All models hit ~96% accuracy purely due to class imbalance; F1/Recall/AUC are the real performance indicators
  • Rolling backlog is the top urgency signal - Operational context outweighs static service-type rules
  • Prescriptive actions: Deploy the hybrid priority score, pre-position resources before the July-August surge, implement weather-triggered surge protocols, use ward signals to geo-allocate wildlife teams
  • Next steps: Apply SMOTE + threshold tuning to improve recall, explore LightGBM and stacking ensembles, build a real-time 311 priority dashboard, expand to multi-class urgency levels
